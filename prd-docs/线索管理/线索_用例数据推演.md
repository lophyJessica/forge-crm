## 3. 推演案例

### 案例 1：正常流程 — 手动录入 → AI评分 → 转客户

> **目标**：验证主流程的状态流转、AI评分触发、转客户后CRM+ERP联动

**场景**：销售主管手动录入一条线索，AI评分≥80自动分配张三，张三跟进后转客户

**推演步骤**：

| 步骤 | 操作人 | 操作 | 数据变化 | 验证点 |
|------|--------|------|---------|--------|
| 1 | 王五 | 手动创建线索，填写必填字段，点击提交 | `线索编号 = LEAD20260717-0001（新生成）` `线索状态：（无）→ PENDING_ASSIGN` `AI评分 = 系统计算中...` | 提交后立即触发AI评分 |
| 2 | 系统 | AI评分完成 | `AI评分 = 85`（来源=官网15/15 + 行业=IT 18/20 + 历史转化 20/25 + 活跃度 12/15 等） | 85分 ≥80，进入自动分配 |
| 3 | 系统 | 自动分配 | `线索状态：PENDING_ASSIGN → ASSIGNED` `归属人 = 张三` `分配时间 = 2026-07-17 10:00:00` | 自动分配给最优销售 |
| 4 | 张三 | 添加跟进记录（电话沟通） | `线索状态：ASSIGNED → FOLLOWING` `最近跟进时间 = 2026-07-17 10:30:00` | 首次跟进 |
| 5 | 张三 | 判定有效，发起转客户 | `线索状态：FOLLOWING → CONVERTED` `CRM客户快照创建，ERP建档通知发送` | 转客户后不可逆 |

**状态快照**：

| 步骤 | 状态 | AI评分 | 归属人 | 最近跟进 |
|------|------|--------|--------|---------|
| 步骤1后 | PENDING_ASSIGN | — | 无 | 无 |
| 步骤2后 | PENDING_ASSIGN | 85 | 无 | 无 |
| 步骤3后 | ASSIGNED | 85 | 张三 | 无 |
| 步骤4后 | FOLLOWING | 85 | 张三 | 2026-07-17 10:30 |
| 步骤5后 | CONVERTED | 85 | 张三 | 2026-07-17 10:30 |

**走查清单**：
- [ ] AI评分是否在提交后立即触发（不是异步延迟）
- [ ] 自动分配的目标销售是否正确（最优而非随机）
- [ ] 首次跟进后状态是否从ASSIGNED变为FOLLOWING
- [ ] 转客户后状态为终态CONVERTED，不可回退
- [ ] 转客户时是否同步通知ERP建档

**结论**：✅ / ❌

### 案例 2：超时回收 — 分配后48h无人跟进

> **目标**：验证超时回收的时间边界和状态回退

**场景**：线索分配给李四后，48小时没有任何跟进记录，系统自动回收

**推演步骤**：

| 步骤 | 操作人 | 操作 | 数据变化 | 验证点 |
|------|--------|------|---------|--------|
| 1 | 王五 | 分配线索给李四 | `线索状态 = ASSIGNED` `归属人 = 李四` `分配时间 = 2026-07-17 10:00:00` | |
| 2 | 系统 | 距分配已过48h（7/19 10:00），无跟进记录 | `线索状态：ASSIGNED → PENDING_ASSIGN` `归属人 = （清空）` | 48h精确到分钟 |

**边界验证**：

| 场景 | 距分配时间 | 是否有跟进 | 预期结果 |
|------|-----------|-----------|---------|
| 刚好48h，无跟进 | 48h00min | 否 | **回收** |
| 47h59min，无跟进 | 47h59min | 否 | 不回收（差1分钟） |
| 超过48h，但已有跟进 | 50h | 是 | 不回收 |

**走查清单**：
- [ ] 回收时机是否精确到分钟（不是按天批量）
- [ ] 回收后归属人是否完全清空
- [ ] 回收后状态是否回到PENDING_ASSIGN而非删除线索
- [ ] 回收不产生新的分配时间（下次分配时重新记录）

**结论**：✅ / ❌

### 案例 3：放弃 + 公海保护 — 7天内不可再认领

> **目标**：验证放弃流程和公海保护期规则

**场景**：张三跟进后判定无效，填写理由放弃。同一天李四想从公海认领，系统阻断

**推演步骤**：

| 步骤 | 操作人 | 操作 | 数据变化 | 验证点 |
|------|--------|------|---------|--------|
| 1 | 张三 | 放弃线索，填写原因"客户已使用竞品" | `线索状态：FOLLOWING → ABANDONED` `放弃原因 = 客户已使用竞品` | 放弃必填原因 |
| 2 | 李四 | 在公海看到该线索，点击认领 | **阻断**，提示"该线索7天内不可认领（放弃日期：2026-07-17）" | 7天保护期 |
| 3 | 系统 | 7天后（7/24 10:00） | 线索在公海变为可认领状态 | 精确到天 |

**边界验证**：

| 场景 | 距放弃时间 | 预期结果 |
|------|-----------|---------|
| 放弃后第6天 | 6天 | 阻断 |
| 放弃后第7天 | 7天整 | 允许认领 |
| 放弃后第8天 | 8天 | 允许认领 |
| 不同销售（非放弃人）放弃当天尝试 | 0天 | 阻断（7天保护对所有人一致） |

**走查清单**：
- [ ] 放弃是否必须有原因（不能为空）
- [ ] 7天保护期是否对所有销售生效（非仅限制放弃人）
- [ ] 7天后认领是否记录新的分配时间
- [ ] 认领后的状态流转是否与正常分配一致

**结论**：✅ / ❌

### 案例 4：重复线索阻断

> **目标**：验证去重逻辑

**场景**：已存在线索的手机号为13800010001，销售主管再次录入相同手机号

**推演步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1 | 王五录入新线索，手机号=13800010001，点击提交 | **阻断**，提示"该手机号已存在线索 LEAD20260717-0001（强盛科技有限公司），请勿重复录入" |

**边界验证**：

| 场景 | 预期结果 |
|------|---------|
| 手机号相同，邮箱不同 | 阻断 |
| 邮箱相同，手机号不同 | 阻断 |
| 手机号和邮箱都不同，公司名相同 | 不阻断（不同联系人） |

**走查清单**：
- [ ] 阻断提示是否包含已存在线索的编号和公司名
- [ ] 去重是实时校验而非保存后弹窗

**结论**：✅ / ❌
